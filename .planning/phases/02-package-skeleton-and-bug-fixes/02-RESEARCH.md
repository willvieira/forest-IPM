## Summary

Phase 2 converts the existing collection of standalone R scripts into a properly structured R package that passes `R CMD check` with zero errors and zero warnings. The package already has source code in `R/` and `src/` but lacks `DESCRIPTION`, `NAMESPACE`, and any formal package infrastructure. Three concrete bugs must be fixed simultaneously: (1) the C++ eigenvalue solver uses a symmetric-matrix solver on an asymmetric IPM kernel, producing incorrect results; (2) `init_pop()` relies on a global `exists('fct')` check instead of a function-scoped variable; (3) deprecated `purrr::as_vector()` calls will trigger warnings/errors on purrr >= 1.2.0.

The R package infrastructure work is well-understood and tool-assisted — `devtools` and `usethis` handle the mechanical parts (DESCRIPTION skeleton, NAMESPACE generation, Rcpp registration). The main care areas are: correctly declaring all tidyverse `@importFrom` symbols that appear in `params.R`, handling the "no visible binding for global variable" NOTE for column names used in dplyr/tidyr verbs, and importing the pipe operator (`%>%` or switching to native `|>`). The eigenvalue bug requires understanding that `Eigen::EigenSolver` returns complex eigenvalues even for real matrices — the fix must extract the real parts after confirming imaginary parts are negligible.

**Primary recommendation:** Create DESCRIPTION and NAMESPACE via devtools/usethis, fix all three bugs in a single wave, then run `devtools::check()` iteratively until zero errors/warnings. The order matters: fix BUG-03 first (simplest), then BUG-02, then BUG-01 (requires C++ recompile), then tackle PKG-02/03/04/05 as package infrastructure.

### Pattern 5: BUG-01 — General EigenSolver for Non-Symmetric Matrices
**What:** Replace `SelfAdjointEigenSolver` (symmetric only) with `EigenSolver` (general)
**When to use:** IPM kernel K = P + F is not symmetric; symmetric solver produces incorrect results

The challenge: `EigenSolver::eigenvalues()` returns `VectorXcd` (complex vector, `std::complex<double>`). For a real non-negative matrix (IPM kernel), the dominant eigenvalue (lambda) is real by Perron-Frobenius, but we must handle the complex type at the C++/R interface.

Strategy: Return the real parts of eigenvalues as `VectorXd` (discarding imaginary parts). This is valid for IPM kernels where imaginary parts are numerically zero for the dominant eigenvalue.

```cpp
// Source: Eigen documentation + RcppEigen interface patterns
#include <RcppEigen.h>

// [[Rcpp::depends(RcppEigen)]]

using Eigen::Map;
using Eigen::MatrixXd;
using Eigen::VectorXd;
using Eigen::EigenSolver;

// [[Rcpp::export]]
VectorXd getEigenValues(Map<MatrixXd> M) {
    EigenSolver<MatrixXd> es(M, /* computeEigenvectors = */ false);
    // eigenvalues() returns VectorXcd (complex); extract real parts
    return es.eigenvalues().real();
}
```

This preserves the original R-level interface — `getEigenValues()` still returns a real-valued `VectorXd` — so no downstream calling code needs to change. The dominant eigenvalue (lambda) is `max(getEigenValues(K))`.

Verification: `max(getEigenValues(K))` should match `Re(base::eigen(K)$values[1])` within floating point tolerance.

### Pattern 6: BUG-02 — Remove `exists('fct')` Global State Check
**What:** `init_pop()` uses `exists('fct')` to check whether the while-loop ran at all, which inspects global environment state rather than local function scope
**When to use:** The fix is to initialize `fct` before the loop so the post-loop branch is always safe

```r
# BEFORE (buggy — exists() checks global env by default):
while(abs(diff_N) > accuracy) {
  fct = runif(1, Min, Max)
  ...
}
if(exists('fct')) {
  N_out <- dbh_den * fct
} else {
  N_out <- dbh_den
}

# AFTER (correct — initialize fct = 1 before loop):
fct <- 1  # identity scaling; if loop never runs, N_out = dbh_den * 1 = dbh_den
while(abs(diff_N) > accuracy) {
  fct <- runif(1, Min, Max)
  ...
}
N_out <- dbh_den * fct
```

This is equivalent behavior: if the initial density is already within `accuracy` of `N`, the loop never runs, `fct` stays 1, and `N_out = dbh_den * 1 = dbh_den` — matching the original `else` branch.

### Pattern 7: BUG-03 — Replace `as_vector()` with `unlist()`
**What:** `purrr::as_vector()` was soft-deprecated in purrr 1.0.0 and is fully deprecated in purrr 1.2.0 (2025), generating warnings in `R CMD check`
**When to use:** Both call sites in `params.R` must be replaced

```r
# BEFORE:
pivot_wider(names_from = 'par') |>
  as_vector()

# AFTER:
pivot_wider(names_from = 'par') |>
  unlist(use.names = TRUE)
```

`unlist()` on a single-row tibble produces a named numeric vector — exactly what `as_vector()` was doing. The `use.names = TRUE` preserves parameter names (e.g., `r`, `Beta`, `Lmax`, etc.) used as vector indices throughout the IPM code.

### Anti-Patterns to Avoid
- **Writing NAMESPACE by hand:** Roxygen2 generates and manages NAMESPACE; any manual edits are overwritten by `devtools::document()`. Put all import declarations in R code as `@importFrom` tags.
- **Using `@import dplyr`** (importing whole namespace): Import only specific functions with `@importFrom dplyr filter group_by ...` — whole-namespace imports cause R CMD check warnings about masking.
- **Leaving `as_vector()` in code:** purrr 1.2.0 (2025) fully deprecated it; it will trigger warnings in `R CMD check` immediately.
- **Using `exists()` without `inherits = FALSE`:** `exists('x')` walks up the environment chain to global; in a package, this makes function behavior depend on user's global environment state.
- **`SelfAdjointEigenSolver` on non-symmetric matrix:** Produces mathematically wrong eigenvalues silently — no error is raised, but results are incorrect.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| NAMESPACE generation | Custom script to write importFrom lines | `devtools::document()` → roxygen2 | roxygen2 tracks all `@importFrom` tags across all R files; manual maintenance breaks on first edit |
| Rcpp function registration | Manually write `.Call` dispatchers | `// [[Rcpp::export]]` + `devtools::document()` | Rcpp compileAttributes() generates `RcppExports.R` and `RcppExports.cpp` automatically |
| "No visible binding" suppression | `globalVariables()` list | `.data` pronoun from rlang | `globalVariables()` is a blunt instrument; `.data` is precise and documents intent |
| Package dependency declaration | Hand-edit DESCRIPTION Imports | `usethis::use_package("pkg")` | Ensures correct field placement and avoids syntax errors |
| C++ build configuration | Custom Makevars from scratch | Let `Rcpp.package.skeleton` / default flags handle it | Default Rcpp flags handle C++11/14/17 and linking correctly on Linux/macOS |

**Key insight:** The R package toolchain (devtools/usethis/roxygen2/Rcpp) handles 90% of the mechanical work. The real effort in this phase is inventory work (finding all symbols that need `@importFrom`) and the three targeted bug fixes.

## Common Pitfalls

### Pitfall 1: Forgetting to Run `devtools::document()` After Adding `@importFrom`
**What goes wrong:** `R CMD check` reports "undefined symbol" or function not found even though code appears correct
**Why it happens:** NAMESPACE is not regenerated automatically; `@importFrom` tags in R files are just comments until `document()` processes them
**How to avoid:** Run `devtools::document()` after any change to roxygen tags; run `devtools::check()` to validate the full state
**Warning signs:** NAMESPACE file missing `importFrom(pkg, fn)` entries despite `@importFrom` being in source

### Pitfall 2: `useDynLib` Missing from NAMESPACE
**What goes wrong:** `devtools::load_all()` silently loads but `getEigenValues()` is not found; error: "could not find function"
**Why it happens:** Without `useDynLib(forestIPM, .registration=TRUE)` in NAMESPACE, the C shared library is not loaded
**How to avoid:** Add `#' @useDynLib forestIPM, .registration = TRUE` and `#' @importFrom Rcpp sourceCpp` to one R file, then run `devtools::document()`
**Warning signs:** `getEigenValues` works in RStudio (which calls `load_all` differently) but fails after `install_github`

### Pitfall 3: `EigenSolver` Complex Return Type Breaks R Interface
**What goes wrong:** `getEigenValues()` returns a complex matrix in R, not a numeric vector; downstream `max()` calls fail
**Why it happens:** `EigenSolver::eigenvalues()` returns `VectorXcd` (`std::complex<double>`); Rcpp marshals this as a complex matrix in R
**How to avoid:** Call `.real()` on the result before returning: `es.eigenvalues().real()` returns `VectorXd`
**Warning signs:** R-level result is a complex object rather than a numeric vector

### Pitfall 4: `as_vector()` Deprecation Warning Elevates to Error
**What goes wrong:** `R CMD check` reports a warning about deprecated function; if `--as-cran` flag is used, warnings = errors
**Why it happens:** purrr 1.2.0 (2025) fully deprecated `as_vector()`; it now emits a deprecation signal on every call
**How to avoid:** Replace all `as_vector()` calls with `unlist(use.names = TRUE)` before running check
**Warning signs:** `R CMD check` output mentions "deprecated function" in purrr

### Pitfall 5: Column Name "No Visible Binding" NOTEs Become Warnings Under `--as-cran`
**What goes wrong:** NOTEs from bare column names in dplyr verbs appear as warnings in `R CMD check`
**Why it happens:** `filter(par != 'lp__')` — R CMD check sees `par` as a potential global variable reference
**How to avoid:** Use `.data$par` for all column references in data masking contexts; import `.data` from rlang
**Warning signs:** `R CMD check` output lists variable names like `par`, `value`, `vr`, `iter`, `name` as having "no visible binding"

### Pitfall 6: `exists('fct')` Passes in Interactive Session, Fails in Package
**What goes wrong:** Function behavior depends on whether user has `fct` in their global environment
**Why it happens:** `exists()` with default `inherits = TRUE` walks up to global environment; if user ran `fct <- 5` earlier, the branch behavior changes
**How to avoid:** Initialize `fct <- 1` before the while-loop; remove `exists()` check entirely
**Warning signs:** `init_pop()` returns different results in clean vs. dirty R sessions

### Pitfall 7: `params.R` Uses `%>%` Without Declaring It
**What goes wrong:** `R CMD check` reports "no visible binding" or "could not find function '%>%'"
**Why it happens:** `%>%` cannot be called with `::` syntax; it must be imported via `@importFrom magrittr %>%`
**How to avoid:** Add `#' @importFrom magrittr %>%` to the imports block OR migrate to native `|>` and remove magrittr from Imports
**Warning signs:** `R CMD check` error about `%>%` not being found or imported
