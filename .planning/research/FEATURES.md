# Feature Landscape

**Domain:** Scientific R package — Bayesian hierarchical IPM for tree population dynamics
**Researched:** 2026-02-25
**Confidence note:** R package infrastructure standards (CRAN policies, r-pkgs.org) are HIGH confidence from training data — these are stable, well-documented standards. Scientific reproducibility patterns for Bayesian/large-data packages (arrow, tidybayes, ipmr, popdemo) are MEDIUM confidence, cross-checked against known package designs. ML decoder feasibility is LOW confidence — no prior art exists for this exact pattern in ecology.

## Table Stakes

Features users expect in any published R package. Missing = package feels incomplete, CRAN rejects, or users cannot reproduce results.

### Package Infrastructure

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| `DESCRIPTION` file (Title, Version, Authors, License, Depends, Imports, Suggests) | CRAN requirement; `devtools::install_github()` will fail without it | Low | Must pass `R CMD check --as-cran`; License field must match actual LICENSE file |
| `NAMESPACE` file (generated by roxygen2) | Required for `library(forestipm)` to work; controls what is exported | Low | Never edit by hand; always regenerate via `devtools::document()` |
| roxygen2 docstrings on every exported function | Users cannot discover the API without `?function_name` working; CRAN requires Rd files | Medium | Each exported function needs `@param`, `@return`, `@export`, `@examples` tags |
| `R CMD check` passes with 0 errors, 0 warnings | CRAN hard requirement; GitHub Actions CI enforces this | Medium | Rcpp + RcppEigen compilation often triggers NOTE on Windows; must suppress or document |
| At minimum one vignette | CRAN does not mandate vignettes but users expect a worked example for scientific packages; rOpenSci requires it | Medium | `vignette("getting-started")` with `run_ipm()` end-to-end example is sufficient for v1 |
| `README.md` with install instructions and minimal example | GitHub landing page; first thing users see | Low | `devtools::install_github("user/forest-IPM")` + one code snippet showing `run_ipm()` output |
| `NEWS.md` changelog | CRAN convention; users need to track breaking changes | Low | Start at v0.1.0; record breaking changes per version |
| LICENSE file matching DESCRIPTION | CRAN hard requirement | Low | MIT or GPL-3 are standard; choose based on whether downstream use should be open |
| `testthat` test suite (>=3e style) | Prevents regressions; CI catches breaks on every commit; rOpenSci peer review requires tests | Medium | Must cover: parameter loading, kernel assembly, lambda computation, init_pop edge cases |
| `usethis`/`devtools` compatible structure | Developers expect to `devtools::load_all()` and `devtools::check()` | Low | Standard directory layout: `R/`, `src/`, `tests/testthat/`, `vignettes/`, `man/` |

### Rcpp / C++ Integration

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| `src/` directory properly registered for Rcpp compilation | `RcppEigen` eigen solver must compile on install; `useDynLib` in NAMESPACE | Low | Already have `eigen.cpp`; needs `Rcpp::compileAttributes()` run to generate wrappers |
| Cross-platform compilation support (Linux + macOS) | PROJECT.md scopes out Windows initially but macOS/Linux are required | Medium | RcppEigen header availability varies; pin RcppEigen version in DESCRIPTION |
| `LinkingTo: Rcpp, RcppEigen` in DESCRIPTION | Required for Rcpp compilation to find headers | Low | Easy to add; standard Rcpp package pattern |

### Data Access

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Parameter fetching without local 15 GB dataset | Core value proposition from PROJECT.md; without this, package is unusable for most researchers | High | This IS the hardest part of the project |
| Clear error messages when network is unavailable | Users on HPC clusters with no internet need informative failures, not cryptic RDS path errors | Low | `tryCatch` wrapping all HTTP calls with user-facing message |
| Cacheing fetched parameters locally | Fetching 15 GB on every call is a non-starter; users need the parameters to persist across sessions | Medium | `tools::R_user_dir("forestipm", "cache")` is the CRAN-blessed cache location |

### Scientific Correctness

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Exported `run_ipm()` high-level function | Researchers should not need to know internal function call order; a single entry point is expected | Medium | Wraps: getPars_sp → mkKernel → lambda via RcppEigen |
| S3 class for kernel object (`ipm_kernel`) with `print()` and `summary()` | Users expect typed objects that report themselves sensibly | Low | `print.ipm_kernel` should report species, n mesh points, lambda estimate |
| Input validation with informative error messages | Scientific users will pass wrong species codes, out-of-range climate values; silently wrong results are worse than errors | Medium | Validate species against supported list, climate values against training range |

## Differentiators

Features that distinguish an excellent scientific package from a merely functional one. Not expected by default, but valued by reviewers, collaborators, and power users.

### Developer Experience

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| `pkgdown` website (GitHub Pages) | Makes function reference and vignettes searchable; links from README; dramatically improves discoverability | Low | `usethis::use_pkgdown_github_pages()` automates deployment; reference + articles structure |
| Multiple vignettes (getting-started, uncertainty propagation, sensitivity analysis) | Addresses the three most common scientific workflows researchers will attempt | Medium | Vignette 2: drawing from full posterior (4000 draws). Vignette 3: elasticity/sensitivity patterns |
| GitHub Actions CI on push (R-CMD-check across R versions and OS) | Signals package health to collaborators; catches Rcpp regression on macOS vs Linux silently | Medium | `usethis::use_github_action("check-standard")` gives the standard matrix |
| Code coverage reporting (covr + codecov) | Shows test completeness; reviewers and collaborators check this | Low | `usethis::use_coverage()` + codecov badge in README |
| `lifecycle` badges on experimental/deprecated functions | Signals stability; important for a package where the ML decoder is experimental | Low | `lifecycle::badge("experimental")` in roxygen; import lifecycle in DESCRIPTION |
| Contributor guide (`CONTRIBUTING.md`) | Signals openness to collaboration; required for rOpenSci submission | Low | Document how to add a new species, how to run tests |

### Data Access Patterns

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Lazy/partial Parquet reads via `arrow` | Fetching only parameters for the requested species+draw instead of full 15 GB is critical for usability | High | `arrow::open_dataset()` on remote URL with `filter(species == sp)` before `collect()` — requires HTTP range request support on the hosting platform |
| Tiered access: known plots (Parquet) vs unobserved sites (ML decoder) | Extends the package from "reproduce existing results" to "predict anywhere" — a genuine scientific contribution | Very High | ML decoder for spatial random effects is novel; needs separate development and validation vignette |
| Parameter version pinning | Reproducibility requires that a script run in 2027 uses the same parameters as in 2026 | Medium | Version parameter files (e.g., `v1.0/` path prefix in cloud storage); `get_param_version()` helper |
| Offline/bundled minimal dataset for examples and tests | Tests cannot depend on network; CRAN requires examples to run in < 5 seconds | Medium | Bundle posterior means for 1-2 species as internal `sysdata.rda`; tests use this, not cloud |

### Scientific Reproducibility

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Full posterior draw access (4000 draws) for uncertainty propagation | Posterior mean alone is scientifically incomplete; lambda uncertainty matters for coexistence analysis | Medium | `getPars_sp(method = "random", n_draws = 100)` returning a list of parameter sets |
| `set.seed()` aware random draw method | Reproducibility of stochastic IPM runs | Low | Document that `getPars_sp(method = "random")` is affected by RNG state |
| Session info logging in vignettes | Readers can reproduce the exact computational environment | Low | `sessioninfo::session_info()` at end of every vignette |
| DOI citation (`CITATION` file + `citation("forestipm")`) | Scientific packages must be citable; journals require it | Low | `usethis::use_citation()` generates `inst/CITATION`; link to the accompanying paper |
| Companion paper reference in DESCRIPTION `Description:` field | Reviewers look for the published methodology | Low | Add `See <doi:10.xxxx/...> for the underlying model.` to Description |

### API Design

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| S3 classes for all core objects (`ipm_kernel`, `ipm_population`, `ipm_params`) | Enables method dispatch: `plot.ipm_kernel()`, `summary.ipm_population()` | Medium | S3 is idiomatic R; avoids users inspecting list structure directly |
| `plot.ipm_kernel()` method wrapping matrix_image.R | Researchers need to visually inspect kernels; currently only available in matrix_image.R | Low | Should produce heatmap of K matrix with size axes labeled in mm DBH |
| Pipe-friendly API | Modern R users expect `|>` to work through the workflow | Low | Ensure functions return their primary object (not invisible NULL); avoid side effects as primary output |
| `supported_species()` helper function | Users need to know which of the 31 species are available without reading docs | Low | Returns a data.frame with species_id, common name, model variants available |
| Climate input validation against model training range | Prevents out-of-distribution extrapolation silently producing biologically impossible results | Medium | Warn (not error) when Temp/Prec outside training range; document training ranges |

## Anti-Features

Features to deliberately NOT build in v1 — they add complexity without proportionate value at this stage.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Shiny app / interactive GUI | PROJECT.md explicitly out-of-scope; GUI requires separate maintenance burden and adds no scientific value for programmatic users | Programmatic R API only; add pkgdown site for documentation |
| Fitting new species / parameter estimation | Requires Stan/MCMC infrastructure; 10x the complexity of the consumption package; fundamentally different user | Separate package or scripts; document that this package consumes posteriors, not fits them |
| Real-time climate data fetching (API calls to PRISM, ERA5, etc.) | External API dependency = fragility; climate APIs require authentication, have rate limits, change formats; user-provided climate is more reproducible | Document the expected climate input format; provide example climate values for common locations |
| Windows binary support in v1 | RcppEigen compilation on Windows requires Rtools and is a known source of CI complexity; PROJECT.md defers this | Linux + macOS only; add `SystemRequirements: C++ compiler` to DESCRIPTION; document Windows limitation clearly |
| CRAN submission in v1 | CRAN submission overhead (manual review queue, stricter NOTE policies) is premature before the cloud data access pattern is proven stable | GitHub-only install via `devtools::install_github()`; pursue CRAN only after v1 stabilizes |
| Custom memory-mapped parameter format (HDF5, custom binary) | Arrow + Parquet already solves partial reads with HTTP range requests; inventing a format adds maintenance with no user benefit | Parquet via `arrow` package; standard format means users can inspect files independently |
| Population genetics / phylogenetic features | Out of domain; the model is purely demographic | Stay in demographic IPM space |
| Multi-species community IPM in v1 | Coexistence analysis (already partially in `simulations/`) requires a community-level kernel that multiplies complexity | Single-species IPM first; community extension is a future milestone once core API is stable |
| Automatic plot discovery (matching lat/lon to known plots) | Requires spatial join against plot database; adds spatial dependencies and significant complexity | User either provides known `plot_id` OR provides coordinates for ML decoder path; no magic matching |

## Feature Dependencies

```
DESCRIPTION + NAMESPACE
  └── roxygen2 docs (depends on: DESCRIPTION for package imports)
        └── exported functions (depends on: roxygen2 for @export)
              └── run_ipm() high-level API (depends on: exported functions)
                    └── vignettes (depends on: run_ipm() working end-to-end)
                          └── pkgdown site (depends on: vignettes + roxygen2 docs)

Rcpp registration (compileAttributes)
  └── RcppEigen lambda solver (depends on: Rcpp registration)
        └── run_ipm() (depends on: lambda solver)

Cloud Parquet parameter storage
  └── arrow lazy reads (depends on: Parquet files hosted with HTTP range request support)
        └── getPars_sp() cloud version (depends on: arrow lazy reads)
              └── bundled sysdata.rda (offline fallback for tests/examples, depends on: knowing final parameter format)
                    └── testthat suite (depends on: bundled sysdata.rda — tests must not require network)
                          └── GitHub Actions CI (depends on: testthat suite passing)

S3 class ipm_params (depends on: getPars_sp() cloud version)
S3 class ipm_kernel (depends on: mkKernel returning classed object)
S3 class ipm_population (depends on: projection simulation returning classed object)

plot.ipm_kernel() (depends on: S3 class ipm_kernel)
summary.ipm_population() (depends on: S3 class ipm_population)

ML decoder for unobserved sites (depends on: ipm_params S3 class; independent of Parquet path)
Parameter version pinning (depends on: cloud Parquet storage)

CITATION file (depends on: companion paper DOI existing)
```

## MVP Recommendation

For v1 (GitHub-installable, usable by collaborators), prioritize:

1. `DESCRIPTION`, `NAMESPACE`, `LICENSE`, `NEWS.md` — no package without these
2. Rcpp registration + `R CMD check` passing — compilation must work on install
3. roxygen2 docs on all exported functions — minimum discoverability
4. Bundled `sysdata.rda` with posterior means for 2 species — enables tests and examples without network
5. `testthat` suite covering kernel assembly + lambda + `init_pop` edge cases — regression safety net
6. Cloud Parquet + `arrow` lazy reads replacing local RDS path in `getPars_sp()` — the core value proposition
7. `run_ipm()` high-level function + S3 class for kernel — the user-facing API
8. One getting-started vignette — minimum documentation for a scientific package
9. `README.md` with install + quick example — GitHub landing page

Defer to v2:
- `pkgdown` site: low effort but not blocking scientific use
- Multiple vignettes (uncertainty, sensitivity): important but after core API stabilizes
- `lifecycle` badges: premature until API is stable
- ML decoder: HIGH complexity, novel research contribution; separate milestone
- Parameter version pinning: needed before first external paper cites the package
- Full posterior draw access (n_draws > 1): needed for uncertainty but adds API complexity
- CRAN submission: after v1 proves stable on GitHub

## Sources

Note: Web search and WebFetch were unavailable during this research session. The following sources informed these findings via training knowledge. All are stable, well-documented standards with HIGH confidence.

- **r-pkgs.org (2e)** (Wickham & Bryan) — canonical reference for R package structure, DESCRIPTION, NAMESPACE, roxygen2, testthat, vignettes, pkgdown
- **CRAN Repository Policy** (https://cran.r-project.org/web/packages/policies.html) — mandatory requirements for DESCRIPTION fields, `R CMD check` policy, examples runtime limits
- **rOpenSci Packages Guide** (https://devguide.ropensci.org/) — peer-review requirements including vignettes, tests, README, CITATION, lifecycle badges
- **`arrow` R package docs** (https://arrow.apache.org/docs/r/) — lazy Parquet reads, `open_dataset()` with remote URLs, HTTP range request behavior
- **`ipmr` package** (Levin et al. 2021, Methods in Ecology and Evolution) — the primary prior art for R-packaged IPMs; architecture reference for kernel objects and S3 methods
- **`popdemo` package** — prior art for population matrix model S3 classes in R
- **`tidybayes` package** — prior art for Bayesian posterior access patterns (draw methods, tidy output)
- **`tools::R_user_dir()`** — CRAN-blessed cache location (R >= 4.0.0); confirmed stable API
